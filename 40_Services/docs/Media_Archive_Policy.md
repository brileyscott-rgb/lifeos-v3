# Media Archive Policy

Status: architecture/scaffold
Created: 2026-07-08
Purpose: define safe handling, storage, linking, backup, and growth monitoring for media files (images, audio, video, documents) captured into the LifeOS pipeline.

## Location

```
/home/lifeos/LifeOS_Media_Archive/
```

This path is OUTSIDE both the canonical LifeOS vault and the capture buffer vault. It is a dedicated media store that can grow large without bloating Obsidian sync or Git repositories.

## Structure

```
LifeOS_Media_Archive/
├── images/
│   ├── YYYY/MM/
│   │   └── {capture_id}_{slug}.{ext}
├── audio/
│   ├── YYYY/MM/
│   │   └── {capture_id}_{slug}.{ext}
├── videos/
│   ├── YYYY/MM/
│   │   └── {capture_id}_{slug}.{ext}
├── documents/
│   ├── YYYY/MM/
│   │   └── {capture_id}_{slug}.{ext}
├── transcripts/
│   ├── YYYY/MM/
│   │   └── {capture_id}_transcript.md
└── README.md
```

Files are organized by type, then year/month for human browsability. Every file is prefixed with its capture ID for source traceability.

## Core Rules

### Do Not Sync Large Videos into Canonical Vault

- Video files stay in `LifeOS_Media_Archive/videos/`
- A Markdown reference note with metadata and a local file link goes into the canonical vault
- Threshold: any video over 10 MB is stored in media archive only
- Small videos (under 10 MB) may be imported into canonical vault attachments folder at operator discretion

### Do Not Sync Large Raw Media into Normal Obsidian Vault

- Raw media files are stored in the media archive, which is NOT an Obsidian vault
- The media archive is NOT synced by Obsidian Sync, Syncthing, or Git
- This prevents large binary files from bloating sync queues and Git history
- The canonical vault may reference media archive files via local absolute or relative symlinks (platform-dependent)

### Markdown Notes Should Link to Local Media

When a Markdown note references a media file, use local file links:

```markdown
![Screenshot of architecture diagram](../../LifeOS_Media_Archive/images/2026/07/cap_20260708_123456_abc123_architecture-diagram.png)

Video: [Watch demo](../../LifeOS_Media_Archive/videos/2026/07/cap_20260708_123457_def456_demo.mp4)
```

The canonical vault and media archive are sibling directories under `/home/lifeos/`, so relative paths work with Obsidian and other Markdown viewers.

### Transcripts Can Be Synced as Markdown

- Transcript files (`.md`) are small and text-based — they may be imported into the canonical vault
- However, all machine-generated transcripts MUST include the `machine_generated_unverified` frontmatter status
- Transcripts imported into the canonical vault go through the normal approval gate

### Whisper Transcripts Must Be Marked machine_generated_unverified

Every transcript generated by Whisper or any automatic speech recognition engine must include:

```yaml
---
transcript_status: machine_generated_unverified
transcript_engine: whisper
transcript_model: whisper-large-v3  # or actual model used
transcript_language: auto
source_media: cap_20260708_123456_abc123_podcast-episode.mp3
---
```

When a human verifies and corrects the transcript:

```yaml
---
transcript_status: human_verified
transcript_engine: whisper
verified_by: lifeos
verified_at: 2026-07-09T14:30:00Z
---
```

Unverified transcripts must not be treated as factual records. Notes citing unverified transcripts must include a caveat.

### Media Backup Strategy

Media files require different backup treatment than Markdown:

| Aspect | Markdown Backup | Media Backup |
|---|---|---|
| Tool | Git + restic | restic only |
| Frequency | Every commit + daily | Daily |
| Retention | Full history | 30 daily, 4 weekly |
| Restore test | Monthly (sample) | Monthly (sample) |
| Off-site | GitHub + restic remote | restic remote only |
| Exclusion | None | Videos over 500 MB optional |

Media files are NOT stored in Git. The media archive is explicitly gitignored.

### Copyright/Source Awareness

When capturing media from external sources:
- Source URL and attribution MUST be preserved in metadata
- Copyright status should be noted if known (personal, public domain, fair use, unknown)
- Personally created media should be marked `source: self`
- Captured web content should preserve the original URL and access date
- Do not re-publish, re-host, or distribute copyrighted media outside personal use

### Storage Growth Monitoring

Media storage must be monitored to prevent disk exhaustion:

- Monitor script checks `LifeOS_Media_Archive/` total size weekly
- Warning threshold: 5 GB
- Critical threshold: 10 GB
- At critical: operator must review and archive/delete old media
- Media archive size included in observability dashboard
- Transcripts count toward the warning threshold but rarely cause issues (text files are small)

### Media Retention

- Media files linked to an approved capture are retained indefinitely (archived)
- Media files from rejected captures: retain 30 days, then delete
- Media files from failed captures: retain 30 days, then delete
- Orphaned media (no capture link found): flag for review, retain 90 days
- Deletion is manual, not automated — operator must approve media cleanup
