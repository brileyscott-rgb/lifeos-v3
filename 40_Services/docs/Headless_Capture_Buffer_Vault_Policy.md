# Headless Capture Buffer Vault Policy

Status: architecture/scaffold
Created: 2026-07-08
Purpose: define the structure, rules, and safety boundaries for the LifeOS Capture Buffer Vault — the isolated workspace where captures are processed, agents draft, and import proposals are prepared before any canonical LifeOS vault write.

## Location

```
/home/lifeos/LifeOS_Capture_Buffer/
```

This path is OUTSIDE the canonical LifeOS vault (`/home/lifeos/10_Vaults/LifeOS/`). It is a separate, headless folder structure optimized for machine processing and agent workspace, not for human Obsidian browsing.

The buffer vault may optionally be opened in Obsidian as a secondary vault for inspection, but it is NOT the primary LifeOS vault and should not be confused with it.

## Core Rules

### Append-Only Raw Capture Rule

- Raw captures in `00_Raw/captures.jsonl` are append-only
- Once written, a raw capture record is never modified
- Processing status updates are written to a separate status field or log, not by editing the raw record
- This preserves the original capture for audit and re-processing

### Unique Filename Rule

- Every file created in the buffer vault uses a capture-ID-prefixed filename
- Format: `{capture_id}_{descriptive_slug}.md`
- Prevents collisions and makes source traceability trivial

### No Canonical Vault Writes from Processors

- Dockerized processors mount the buffer vault and media archive ONLY
- Processors MUST NOT mount any path under `/home/lifeos/10_Vaults/LifeOS/`
- Processors MUST NOT have write access to any path outside their designated mounts
- This is enforced by Docker volume mount configuration, not by convention

### No Direct AI Writes to Canonical LifeOS

- AI agents operate on the buffer vault ONLY
- AI agents MUST NOT be given filesystem access to the canonical vault path
- Even "trusted" agents must go through the approval gate for canonical imports
- Agent permissions are defined per file, not globally

### Buffer-to-Canonical Approval Requirement

- Every import from buffer vault to canonical vault requires:
  1. Complete review packet generated by Import Planner Agent
  2. QA verification passed (QA Verifier Agent)
  3. Human operator explicitly approves via Telegram, dashboard, or CLI
  4. Import executed by import_exporter processor (not by agent, not by n8n, not by MCP)
- No automated, scheduled, or conditional auto-approval in V1

### Sync Conflict Prevention

- The buffer vault is NOT synced by Obsidian Sync, Syncthing, or any real-time sync tool
- It is a local-only, server-side workspace
- The canonical vault IS synced via Git (and optionally Obsidian Sync)
- Never mount the buffer vault inside the canonical vault as a folder or symlink
- Never configure sync tools to watch the buffer vault

### Failed Capture Retention

- Captures that fail processing are moved to `06_Failed/`
- Failed captures are NOT deleted
- Failure reason, stack trace, and retry count are logged
- Failed captures may be manually re-submitted to the queue
- Automatic retry with exponential backoff (configurable cap: 3 retries)

### Duplicate Handling

- Near-duplicate captures are detected by the duplicate_detector processor
- Duplicates are NOT automatically rejected — they are flagged in the review packet
- Operator may reject duplicate (move to `05_Rejected/`) or approve as separate capture
- Duplicate detection uses content similarity, not exact match

### Media Handling

- Media files (images, audio, video, PDFs) are stored in `/home/lifeos/LifeOS_Media_Archive/`
- Buffer vault Markdown files link to media in the media archive
- Media files are NOT stored inside the buffer vault
- Media files are NOT stored inside the canonical vault unless explicitly approved and small
- See `Media_Archive_Policy.md` for full media rules

### Transcript Verification Warning

- All machine-generated transcripts (Whisper, etc.) MUST include frontmatter:
  ```yaml
  transcript_status: machine_generated_unverified
  transcript_engine: whisper
  ```
- Human-verified transcripts may be updated to `human_verified`
- Unverified transcripts must not be cited as factual without verification

### Cleanup/Archive Policy

- `03_Review_Packets/` — remove after 90 days of approval/rejection (packet data in event log)
- `04_Approved_For_Import/` — remove after confirmed successful import + 30 days
- `05_Rejected/` — retain 30 days then optionally archive
- `06_Failed/` — retain 30 days then optionally archive
- `00_Raw/` — processed captures may be trimmed after 90 days (event log retains record)
- Agent workspace drafts — remove after packet generated + 7 days
- Logs — rotate monthly, retain 3 months

### Backup Policy

- The buffer vault is backed up via restic to the primary backup location
- Backup schedule: daily, included in the standard LifeOS backup rotation
- Media archive is backed up separately (larger, less frequent)
- Backup retention: 7 daily, 4 weekly, 3 monthly
- Backup verification: monthly restore test of a sample capture

### What Can and Cannot Sync

**CANNOT sync to phone/desktop Obsidian:**
- The entire buffer vault
- `00_Raw/`, `01_Processed/`, `02_Agent_Workspace/` (these are machine workspaces)
- `03_Review_Packets/` (internal approval artifacts)
- `06_Failed/`, `07_Logs/`

**CAN sync to phone/desktop Obsidian if copied into canonical vault:**
- Approved imported notes (after import into canonical vault, which IS synced)
- Review packet summaries (if operator chooses to create a canonical review log)

The buffer vault is a headless server-side workspace. It is intentionally NOT synced to mobile devices.
